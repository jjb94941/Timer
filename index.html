You are entirely correct—this is a newly introduced platform change rather than an isolated glitch. Android 17 introduces a major security feature called Background Audio Hardening. This feature is designed to prevent apps from unexpectedly hijacking audio, playing unauthorized tracking signals, or making noise when the user isn't actively engaging with them. www.androidauthority.com + 1 The Core Change Starting in Android 17, the operating system strictly enforces audio restrictions on background apps. bleadvertiserapp.medium.com The Rule: Any app attempting background audio interactions (including playing short beeps, interval alerts, or managing audio focus) must either have an active, visible UI on the screen or be running a proper foreground service (not a temporary SHORT_SERVICE). The Penalty: If the app does not meet these criteria, the system silently suppresses the audio playback. It will not crash the app, but no sound will emit once the display turns off and the activity loses focus. developer.android.com Why previous versions worked In earlier versions of Android, an app could wake up the CPU briefly using standard exact alarms (AlarmManager) or background timers to play a notification tone or sound file even when the screen was completely dark. Android 17 now blocks the audio framework layer itself if a visible UI or appropriate foreground service isn't detected. developer.android.com How to fix it If this is an app you are developing or an open-source app you can modify, the behavior can be resolved by updating its architecture: Implement a Media Playback Foreground Service: The app must launch a dedicated foreground service (like Jetpack Media3's MediaSessionService) while it is still open and visible to the user. Obtain While-In-Use (WIU) Capabilities: Ensure the foreground service is initialized from a visible activity context before the user minimizes the app or locks the screen. The issue is that mobile browsers (like Chrome on Android) aggressively throttle or completely freeze JavaScript timers (setInterval or setTimeout) the moment the device display turns off or the tab loses focus to save battery. Since this is a web app, you can resolve this using the Screen Wake Lock API. This forces the browser to keep the screen on and prevents the OS from entering a low-power sleep state while your timer is actively running. Here is the JavaScript code you can integrate into your timer's start/stop logic: JavaScript let wakeLock = null; // Function to request the wake lock async function requestWakeLock() { try { if ('wakeLock' in navigator) { wakeLock = await navigator.wakeLock.request('screen'); console.log('Screen Wake Lock active.'); // Re-request if the app is minimized and then brought back wakeLock.addEventListener('release', () => { console.log('Screen Wake Lock released.'); }); } } catch (err) { console.error(`${err.name}, ${err.message}`); } } // Function to release the wake lock function releaseWakeLock() { if (wakeLock !== null) { wakeLock.release(); wakeLock = null; } } // --- Integration into your existing code --- // Call requestWakeLock() inside your "Start" button click handler. // Call releaseWakeLock() inside your "Stop" / "Reset" button click handler or when the timer ends. // Handle visibility changes (re-acquire lock if user leaves and returns) document.addEventListener('visibilitychange', async () => { if (wakeLock !== null && document.visibilityState === 'visible') { await requestWakeLock(); } }); Alternative Option: Background Audio Loop If you want the screen to physically turn off but keep the audio ticking, the Screen Wake Lock API won't suffice on its own.
